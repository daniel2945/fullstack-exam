# שאלה 2: ארכיטקטורת מערכת אחזור ותחקירים סמנטית

## 1. רכיבי המערכת המרכזיים
המערכת מורכבת מארבעה רכיבים עצמאיים המופרדים בצורה מודולרית:

* **ממשק משתמש (Frontend):** דף אינטרנט הכולל שורת חיפוש פשוטה ותצוגה נוחה של הכתבות שנמצאו.
* **שרת אחורי (Backend):** שרת (ב-Node.js עם Express) שמנהל את הלוגיקה העסקית, מקבל את בקשות החיפוש מהמשתמש ומקשר בין חלקי המערכת.
* **מנוע אחזור סמנטי (Vector Database):** רכיב מבוסס AI המשתמש במודל שפה (Embeddings). הוא הופך טקסטים לנקודות ציון מתמטיות (וקטורים) ומחפש כתבות לפי משמעות והקשר, ולא רק לפי מילים מדויקות.
* **מסד נתונים ראשי (Primary Database):** מסד נתונים NoSQL (MongoDB) שמיועד לאחסן את התוכן המלא של מיליוני הכתבות ואת פרטי המסגרת שלהן (כמו שם הכתב, תאריך פרסום ותגיות) בצורה גמישה ומהירה.

## 2. מימוש השרת (Backend) וזרימת המידע
השרת האחורי ימומש באמצעות **Node.js ו-Express**. הוא משמש כ"מתווך" מרכזי או כ"שער כניסה" (Gateway) שמקשר ומנווט את המידע בין המסך של המשתמש לבין מנוע ה-AI ומאגרי המידע.

### איך המידע זורם דרך השרת בזמן שהמשתמש מחפש כתבה:
1. **קבלת הבקשה:** המשתמש מקליד משפט לחיפוש באתר. השרת מקבל את המשפט הזה (בנתיב API קבוע, למשל: `GET /search`).
2. **פנייה ל-AI (הבנת המשמעות):** השרת לוקח את משפט החיפוש ושולח אותו למנוע החיפוש הסמנטי (ה-AI) כדי שזה יתרגם את המשפט ל"נקודת ציון" מתמטית שמייצגת את המשמעות שלו.
3. **שליפת הכתבות מהמחסן:** בעזרת נקודת הציון הזו, השרת פונה למסד הנתונים, מזהה את הכתבות שהכי קרובות למשמעות המבוקשת, ושולף את הכותרות והטקסט המלא שלהן מתוך ה-MongoDB.
4. **החזרת תשובה:** השרת אורז את הכתבות שנמצאו ומחזיר אותן בצורה מסודרת למסך של המשתמש.

## 3. מסדי הנתונים והשיקולים מאחוריהם
כדי לטפל במיליוני כתבות בצורה מהירה, אנחנו מחלקים את האחסון לשני מאגרים נפרדים, כשכל אחד מהם מתמחה במשהו אחר:

* **מסד הנתונים הראשי (MongoDB):** כאן נשמור את תוכן הכתבות המלא ואת פרטי המסגרת שלהן (כמו תאריך, כותרת ושם הכתב). 
  * *השיקול:* כתבות אינטרנט מגיעות במבנים משתנים (חלקן ארוכות, חלקן קצרות, עם תגיות שונות). MongoDB הוא מסד נתונים מסוג NoSQL שמאפשר גמישות מוחלטת במבנה הנתונים (Schema-less) ומסוגל להחזיק מיליוני מסמכים ולשלוף אותם בשבריר שנייה לפי המפתח שלהם.
* **מסד הנתונים הווקטורי (Vector Database - מנוע ה-AI):** כאן לא נשמור את הכתבות עצמן, אלא רק את "נקודות הציון" המתמטיות (הווקטורים) שה-AI ייצר עבור כל כתבה כדי לייצג את המשמעות שלה.
  * *השיקול:* מסד נתונים רגיל לא יודע לחשב "מרחק בין משמעויות" של מילים. מסד נתונים וקטורי ייעודי (כמו Qdrant או Milvus) בנוי במיוחד כדי לעשות השוואות מתמטיות מהירות בין מיליוני וקטורים ולמצוא את הכתבות שהכי קרובות למשמעות של שאילתת החיפוש.

## 4. דגש על ארכיטקטורה גמישה ומודולרית
הפרדת המערכת לרכיבים עצמאיים (המסך, השרת, ה-AI, והדאטה-בייס) מעניקה לנו גמישות מקסימלית לשדרוגים עתידיים:

* **שדרוג מודל ה-AI:** אם בעתיד נרצה להחליף את מודל השפה במודל AI חדש ומתקדם יותר, נצטרך לעדכן אך ורק את הרכיב האחראי על הפיכת המילים לווקטורים. שאר חלקי המערכת (השרת, האתר ומסד הנתונים הראשי) לא יושפעו וימשיכו לעבוד כרגיל.
* **החלפת מסד הנתונים:** אם נחליט לעבור מ-MongoDB למסד נתונים אחר, השינוי יתבצע רק ברכיב הגישה לנתונים, מבלי שנצטרך לכתוב מחדש את הלוגיקה של החיפוש הסמנטי או את ממשק המשתמש.
